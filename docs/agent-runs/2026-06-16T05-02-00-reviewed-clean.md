## DeepSeek's one finding — dismissed as unreachable

> `dealSaveLabel` parses `"-$10"` as `10`, returning a positive save label when
> the original string indicates a negative value. e.g.
> `dealSaveLabel({ save: "-$10" }, "USD")` → `"Save USD $10.00"`.

Verified against the real data (`dealPackages`, main.tsx:532-572): all three
deals carry a **positive** `saveAmount` (28 / 34 / 18) AND a positive `save`
string (`"Save $28" / "Save $34" / "Save $18"`). `dealSaveLabel` prefers
`deal.saveAmount` and only falls back to string-parsing when `saveAmount` is
absent — which never happens here. The negative-input branch is therefore
**unreachable in practice**; the example input (`save: "-$10"`, no `saveAmount`)
does not exist anywhere in the codebase.

Consistent with the standing rule: *always verify advisor claims against the
real file before applying.* No defensive code added for an input that cannot
occur (would be churn on a polished, clean app shell).
