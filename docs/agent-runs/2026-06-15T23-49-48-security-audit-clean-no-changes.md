## Why this is a "no changes" pass
The marathon rule is to apply *only real fixes* and *not churn already-polished shells*.

~4.9k lines) found no untrusted-input sinks, so the honest result is a clean bill of health.

## What was checked (and why it's safe)
- **Redirect sink** `window.location.href = intent.booking.checkoutUrl` (main.tsx:4440):
  `checkoutUrl` is generated by the local `checkoutUrl()` → `engineUrl(path)` =
  `new URL(path, engineOrigin)`, where `engineOrigin` is a trusted build-time constant
  (`VITE_ZIVO_PLATFORM_ORIGIN` / bridge config / `https://zivosmedia.com`). All path params
  go through `URLSearchParams` (encoded). No DB/user free-text reaches the URL → not an open
  redirect. DeepSeek concurred ("Safe… not an open redirect in this context").
- **postMessage**: none. No `postMessage(` sends and no `window` `message` listeners exist,
  so there is no cross-frame credential/handoff surface to harden (unlike the autorepair
  modules in zivosmedia/zivosoftware).
- **HTML injection**: none. No `dangerouslySetInnerHTML`, `innerHTML`, or `document.write`.
  Dynamic values render as React text children (auto-escaped).
- **SSO/handoff**: `ssoUrl` is `engineUrl(authHandoff + "?...&redirect=" + encodeURIComponent(internalPath))`
  — trusted origin + encoded internal redirect param. `handoffSource` is read from the query
  string but only used for a display label and an allowlisted boolean
  (`=== "zivosmedia" || === "zivo-admin"`), so it's neither an XSS nor a logic risk.

## One noted nit — deliberately NOT changed
`withBookingReference()` returns `${url.pathname}${url.search}` on the relative-path branch,
which drops any `#hash`. No booking/sso/review URL in this app uses a hash fragment, so this
is a theoretical edge case only. DeepSeek's verdict matched: *"not worth changing for current
usage."* Changing it would be speculative churn against a working code path.
